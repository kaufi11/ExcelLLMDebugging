Short answer — what’s wrong
1) Many applicant-score formulas point to the wrong weight cells on the Rate sheet. Any applicant L-cell that is supposed to score Rate!G16..G20 is instead using the wrong Rate!E row (off by 1–2 rows). This appears in Ap1, Ap2 and Ap3.
2) Ap3 L20 uses inequality tests (<>), not equality (=), so its logic is inverted/incorrect.

Why that breaks results
- The wrong Rate!E reference uses the wrong numeric weight (E11..E20 are the intended weights). Scores therefore are calculated from incorrect weights and totals are wrong.
- The <> tests in Ap3 L20 produce wrong multipliers for K20.

How to implement
- Edit each listed cell and paste the corrected formula.
- After fixing, check Summary totals (Summary L8:L10) — they should change to reflect corrected scores.

Prevent this in future (quick tips)
- Use a single reliable lookup rather than hard-coded E references. Example: store the weight next to each preference and use INDEX/MATCH or VLOOKUP to fetch the weight by the preference text. That avoids hard-coded row-number mistakes.
- Keep the same formula structure for all applicants and copy/paste rather than hand-edit cell references.
