Summary of the problems
- Several Ap sheets contain incorrect references to the Rate sheet’s weight column (E11:E20). Some L-cells multiply the wrong E-row (e.g., using E17 when the question text in B points to G19). That misalignment is an off-by-row error and will produce wrong scores if Rate!E values change (they all happen to be 100 now, which hides the bug).
- Ap3 L20 contains a hard-coded literal "784" inside the IF tests (IF(784=2,...)) instead of the K20 cell reference. That always evaluates false and returns wrong results.

Why this matters
- Because L-cells are supposed to multiply the weight that corresponds to the same preference text (Rate!Gxx ↔ Rate!Exx), wrong E references produce incorrect partial scores and therefore wrong totals. The 784 typo breaks that cell completely.

Recommendation — make formulas robust (prevents future mapping mistakes)
- Instead of hard-coding the E-row numbers into each cell, use MATCH/INDEX to find the correct weight by matching the preference text in column B back to Rate column G. That avoids row-number errors when rows are moved or the order is changed.

Example robust formulas
- Proficiency (replace any L16-type formula):
