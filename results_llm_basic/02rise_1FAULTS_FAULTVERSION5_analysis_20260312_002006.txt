I found three concrete errors that will make your totals and savings wrong. Short fixes and why they work below.

1) Total Savings formulas use the wrong subtraction order (sign reversed)
- Problem: the “Total Savings” rows (the row immediately after each class’s “Total” row) use formulas like:
  =SUM(F4 - $D3 * F3)
  That computes (RISE cost - public cost) × participants, so savings come out negative.
- Correct formula: compute (public cost − RISE cost) × participants. For the Dreamweaver example (class row 3 / savings row 5) use:
  =F3 * ($D3 - $E3)
  or equivalently:
  =$D3*F3 - F4
- How to apply: For each class block, replace the formulas in the “Total Savings” row (F:BG for that row) with:
  =<company_participants_cell> * ($D<class_row> - $E<class_row>)
  then copy across columns. Repeat for every class block.
- Why: public cost minus RISE cost per participant is the actual saving per participant; then multiply by participants.

2) One hard-coded wrong multiplication in the MS Project totals (AU49)
- Problem: AU49 currently shows:
  =SUM(609*$E$48)
  (609 is hard-coded and not the AU48 participants cell). That corrupts the class totals.
- Fix: change AU49 to:
  =AU48 * $E$48
  (or =SUM(AU48*$E$48) to match neighboring formula style).
- Why: the pattern for totals is (participants by company) × (RISE cost). Replace the literal 609 with the cell that holds that company’s participants.

3) BK261 (Number of participants per class) was hard-coded
- Problem: BK261 currently contains the value 12 instead of the normal formula:
  =SUM(F261:BG261)
  That will make any aggregates that depend on BK (student days, summary counts) incorrect.
- Fix: set BK261 to:
  =SUM(F261:BG261)
- Why: BK should always be the sum of participant columns F:BG for that class so other class and grand totals stay correct.
